# Rust API Service Template

A production-oriented Rust REST API template built with Axum, Tokio, Diesel Async, PostgreSQL, Valkey, JWT authentication, and a module-first application structure.

The project is intended as a clean starting point for API services that need typed configuration, explicit dependency wiring, request middleware, database-backed persistence, token-backed authentication, and containerized local infrastructure.

## Technology Stack

| Layer | Technology |
|---|---|
| HTTP server | Axum |
| Async runtime | Tokio |
| Middleware | Tower, tower-http |
| Persistence | PostgreSQL |
| Query layer | Diesel, diesel-async |
| Schema migrations | Flyway |
| Token/session state | Valkey, Redis protocol |
| Authentication | JWT access and refresh tokens |
| Password hashing | Argon2 |
| Serialization | Serde |
| Validation | validator |
| Error handling | thiserror, anyhow |
| Logging | tracing, tracing-subscriber |
| Container runtime | Docker, Docker Compose |

### Boundaries

- `core/` owns application bootstrap, runtime configuration, shared state, response/error mapping, middleware, and telemetry.
- `infra/` owns external system adapters such as PostgreSQL pool initialization and Valkey token storage.
- `modules/` owns feature-specific controllers, services, domain DTOs, extractors, guards, and repositories.
- `schema.rs` is generated by Diesel from the active database schema.
- `main.rs` loads configuration, initializes telemetry, builds state and router, then serves HTTP with graceful shutdown.

## Runtime Architecture

`core::app::build_router` composes three route groups under `/api/v1`:

| Route group | Modules | Middleware |
|---|---|---|
| Public | health, auth register/login/refresh | Request context, logging, CORS, timeout, body limit, compression |
| Authenticated | auth me/logout, user lookup | Public middleware plus `require_auth` |
| Admin | user administration | Public middleware plus `require_admin` |

Application dependencies are constructed once in `build_state` and stored in `AppState`:

- PostgreSQL connection pool
- Valkey client-backed token store
- JWT service
- Auth service
- User service
- User repository

## API Contract

Default local base URL:

```txt
http://localhost:8088/api/v1
```

Docker Compose API base URL:

```txt
http://localhost:8089/api/v1
```

Successful responses use a consistent envelope:

```json
{
  "data": {}
}
```

Error responses use:

```json
{
  "code": "VALIDATION_ERROR",
  "message": "Invalid request data."
}
```

### Routes

| Method | Path | Auth | Module | Description |
|---|---|---|---|---|
| `GET` | `/health/live` | Public | health | Liveness check |
| `GET` | `/health/ready` | Public | health | Readiness check with PostgreSQL query |
| `POST` | `/register` | Public | auth | Create a user and issue access/refresh tokens |
| `POST` | `/login` | Public | auth | Authenticate a user and issue access/refresh tokens |
| `POST` | `/refresh` | Public | auth | Issue a new access token from a valid refresh token |
| `GET` | `/me` | Bearer token | auth | Return the current authenticated user |
| `POST` | `/logout` | Bearer token | auth | Blacklist the current access token and revoke the refresh token |
| `GET` | `/users/{id}` | Bearer token | user | Return a user by id |
| `GET` | `/admin/users` | Admin bearer token | user | Return all users |

Request examples are available in:

- `http/health.http`
- `http/auth.http`
- `http/user.http`

## Authentication

The auth module uses short-lived access tokens and long-lived refresh tokens.

- Access and refresh tokens are signed with separate secrets.
- JWT claims include `sub`, `jti`, `role`, `iat`, and `exp`.
- Refresh token identifiers are stored in Valkey with a TTL.
- Refreshing keeps the existing refresh token and returns a new access token.
- Logout blacklists the access token until expiry and removes the refresh token identifier from Valkey.
- Protected routes require `Authorization: Bearer <access_token>`.
- Admin routes require a token whose `role` claim is `ADMIN`.

## Configuration

Configuration is loaded from environment variables. In local development, `.env` is loaded automatically.

Create a local environment file:

```sh
cp .env.example .env
```

| Variable | Default | Description |
|---|---:|---|
| `APP_ENV` | `local` | Runtime environment: `local`, `development`, `staging`, or `production` |
| `APP_HOST` | `0.0.0.0` | HTTP bind host |
| `APP_PORT` | `8088` | HTTP bind port for local runs |
| `APP_REQUEST_TIMEOUT_SECONDS` | `30` | Request timeout applied by Tower middleware |
| `APP_BODY_LIMIT_MB` | `2` | Maximum request body size in MiB |
| `CORS_ALLOWED_ORIGINS` | local frontend origins | Comma-separated CORS allowlist |
| `DATABASE_URL` | required | PostgreSQL connection string |
| `DATABASE_MAX_CONNECTIONS` | `10` | PostgreSQL pool size |
| `VALKEY_URL` | required | Valkey/Redis connection string |
| `JWT_ACCESS_SECRET` | required | Secret used to sign access tokens |
| `JWT_REFRESH_SECRET` | required | Secret used to sign refresh tokens |
| `JWT_ACCESS_TTL_SECONDS` | `900` | Access token lifetime |
| `JWT_REFRESH_TTL_SECONDS` | `2592000` | Refresh token lifetime |
| `RUST_LOG` | `info,tower_http=info,diesel=warn` | Tracing filter |

Production configuration validates that JWT secrets are non-empty, distinct, at least 32 characters, and do not contain the default placeholder value. Production CORS configuration must not include `*`.

## Local Development

Start PostgreSQL and Valkey:

```sh
docker compose up -d postgres valkey
```

Run migrations:

```sh
docker compose run --rm migrate
```

Start the API locally:

```sh
cargo run
```

Check service health:

```sh
curl http://localhost:8088/api/v1/health/live
curl http://localhost:8088/api/v1/health/ready
```

Run the test suite:

```sh
cargo test
```

Run formatting and compile checks:

```sh
cargo fmt --check
cargo check
```

## Docker Compose

Build and run the full stack:

```sh
docker compose up --build
```

The Compose stack includes:

- PostgreSQL
- Valkey
- Flyway migration job
- API service

The API container listens on `8089` by default:

```txt
http://localhost:8089/api/v1
```

The Compose file defaults the API container to `APP_ENV=production`. Provide production-grade JWT secrets before running in production mode, or run with `APP_ENV=local` for local Compose workflows.

For local Compose experiments, run the API with local-mode configuration:

```sh
APP_ENV=local docker compose up --build api
```

## Database

Migrations are stored in `migrations/` and executed by Flyway.

Current migration set:

- `V001__create_users.sql`

The `users` table stores:

- `id` as `BIGINT GENERATED BY DEFAULT AS IDENTITY`
- unique `email`
- hashed `password`
- `first_name`
- optional `last_name`
- constrained `status`
- `created_at` and `updated_at`
- optional `deleted_at` for soft deletion

Regenerate Diesel schema after changing migrations:

```sh
diesel print-schema > src/schema.rs
```

## Observability And Middleware

The router applies cross-cutting middleware globally:

- request body limit
- request timeout
- CORS allowlist
- response compression
- structured request logging
- request context propagation

Tracing is configured during startup through `core::telemetry` and controlled by `RUST_LOG`.

## Development Notes

- Keep route registration inside each feature module controller.
- Keep business rules in services rather than controllers.
- Keep database access behind repository traits.
- Keep infrastructure-specific clients under `infra/`.
- Keep successful responses wrapped in `ApiResponse<T>`.
- Map user-facing failures through `AppError` so status codes and error bodies remain consistent.
