# Rust API Service Template

Rust REST API template built with **Axum**, **Tokio**, **Tower middleware**, **Diesel Async**, **PostgreSQL**, **Flyway SQL migrations**, **Valkey**, and a modular service/repository layout.

## Current Stack

| Layer | Technology |
|---|---|
| HTTP framework | Axum |
| Async runtime | Tokio |
| Middleware | Tower / tower-http |
| ORM / query layer | Diesel + diesel-async |
| Database | PostgreSQL |
| Migration tool | Flyway |
| Migration format | Raw SQL |
| Token/session backend | Valkey, via the Redis protocol |
| Password hashing | Argon2 |
| JWT | jsonwebtoken |
| Logging / tracing | tracing + tracing-subscriber |
| Validation | validator |
| Containerization | Docker / Docker Compose |

## Architecture

The codebase follows a module-oriented structure:

```txt
src/
├── common/
├── core/
│   ├── app.rs
│   ├── app_state.rs
│   ├── config.rs
│   ├── error.rs
│   ├── response.rs
│   └── telemetry.rs
├── infra/
│   ├── diesel/
│   └── valkey/
├── modules/
│   ├── auth/
│   ├── health/
│   └── user/
├── schema.rs
├── lib.rs
└── main.rs
```

Main conventions:

- `AppState` stores shared dependencies explicitly.
- Feature modules own their route setup, handlers, DTOs, service logic, and repository contract where needed.
- `core/app.rs` builds application state and mounts all module routes under `/api/v1`.
- Diesel Async is used for runtime PostgreSQL access.
- Flyway owns schema migration; the Rust app does not run migrations itself.
- Valkey stores refresh-token validity and access-token blacklist entries.
- Errors are centralized through `AppError` and converted into JSON error responses.
- Successful API responses use the shape `{ "data": ... }`.

## API Routes

Default local base URL:

```txt
http://localhost:8088/api/v1
```

| Method | Path | Module | Description |
|---|---|---|---|
| `GET` | `/health/live` | health | Liveness check |
| `GET` | `/health/ready` | health | Readiness check with database query |
| `POST` | `/register` | auth | Create user and issue access/refresh tokens |
| `POST` | `/login` | auth | Authenticate with email/password |
| `POST` | `/refresh` | auth | Issue a new access token from a refresh token |
| `POST` | `/logout` | auth | Blacklist access token and revoke refresh token |
| `GET` | `/me` | auth | Return current authenticated user |
| `GET` | `/users/{id}` | user | Return user by id |

Request examples live in:

- `http/health.http`
- `http/auth.http`
- `http/user.http`

## Environment

Create a local `.env` from the example:

```sh
cp .env.example .env
```

Important variables:

| Variable | Purpose |
|---|---|
| `APP_ENV` | `local`, `development`, `staging`, or `production` |
| `APP_HOST` / `APP_PORT` | HTTP bind address |
| `APP_REQUEST_TIMEOUT_SECONDS` | Request timeout used by Tower middleware |
| `APP_BODY_LIMIT_MB` | Request body limit used by Tower middleware |
| `DATABASE_URL` | PostgreSQL connection string |
| `DATABASE_MAX_CONNECTIONS` | Diesel Async pool size |
| `VALKEY_URL` | Valkey/Redis connection string |
| `JWT_ACCESS_SECRET` / `JWT_REFRESH_SECRET` | JWT signing secrets |
| `JWT_ACCESS_TTL_SECONDS` / `JWT_REFRESH_TTL_SECONDS` | Token TTL values |
| `CORS_ALLOWED_ORIGINS` | Comma-separated allowed origins |
| `RUST_LOG` | tracing filter |

In `production`, JWT secrets must be at least 32 characters, must be different from each other, and must not contain the placeholder value `change-me`.

## Local Development

Start PostgreSQL and Valkey:

```sh
docker compose up -d postgres valkey
```

Run migrations:

```sh
docker compose run --rm migrate
```

Run the API locally:

```sh
cargo run
```

Check health:

```sh
curl http://localhost:8088/api/v1/health/live
curl http://localhost:8088/api/v1/health/ready
```

Run tests:

```sh
cargo test
```

## Docker Compose

Build and run the full stack:

```sh
docker compose up --build
```

The Compose stack includes:

- PostgreSQL
- Valkey
- Flyway migration job
- API service

By default, the API container uses `APP_ENV=production`. Provide non-placeholder JWT secrets before running the API container in production mode, or set `APP_ENV=local` for local Compose experiments.

Example:

```sh
APP_ENV=local docker compose up --build api
```

The default Compose API URL is:

```txt
http://localhost:8089/api/v1
```

## Database And Schema

Migrations live in `migrations/` and are executed by Flyway.

Current migration files:

- `V001__create_users.sql`

Current schema characteristics:

- `users.id` is `BIGINT GENERATED BY DEFAULT AS IDENTITY`.
- `users.email` is unique.
- `users.password` stores the password hash.
- `users.first_name` is required.
- `users.last_name` is optional.
- `users.status` is constrained to `ACTIVE` or `DISABLED`.
- Soft delete timestamp exists via `deleted_at`.

Regenerate Diesel schema after migration changes:

```sh
diesel print-schema > src/schema.rs
```

## Auth Flow

The auth module supports:

- Registering users with `email`, `password`, and `name`.
- Logging in with `email` and `password`.
- Creating access and refresh JWTs.
- Saving refresh-token ids in Valkey.
- Refreshing access tokens while keeping the existing refresh token.
- Logging out by blacklisting the current access token and revoking the refresh token.
- Loading the current user from a Bearer access token.

Current caveat: the database status constraint stores `ACTIVE` / `DISABLED`, while the active-user guard in auth service currently checks for lowercase `active`. Until that is normalized in code, endpoints that verify active status may reject users created by the current register flow.
